# Phase 3A application and presentation layer

## Why this layer exists

Phase 2 separated scientific calculations from the legacy GUI. Phase 3A adds a
stable boundary for future interfaces so they do not need to understand
Tkinter state, pandas-based legacy adapters, plotting, spreadsheets, output
folders, or dialogs.

```text
UI / CLI / Notebook
        |
        v
request contract
        |
        v
application service
        |
        v
scientific core
        |
        v
result contract
    /          \
tables        plots
    \          /
     exporters
```

The packages have distinct responsibilities:

- `iclotspython.core` owns deterministic scientific calculations.
- `iclotspython.application` owns requests, validation, workflow
  orchestration, progress, cancellation, and error translation.
- `iclotspython.presentation` owns immutable UI-facing results and
  conversion to table-, plot-, and export-neutral structures.

The application layer calls the core; it does not restate formulas or apply
undocumented corrections.

## ROI request lifecycle

`ROIAccumulationRequest` accepts preloaded arrays, selected named channels,
thresholds, calibration, an explicit `RGB` or `BGR` convention, optional
labels/time values, and non-personal workflow metadata. Paths are not part of
the scientific request.

Validation:

1. requires at least one equally shaped `(height, width, 3)` array;
2. requires an explicit `RGB` or `BGR` convention;
3. validates selected channel names and per-channel thresholds;
4. validates finite positive calibration;
5. validates label, time, and source-identifier counts;
6. preserves supplied non-monotonic times with a warning;
7. copies frames into read-only arrays before execution.

Future loaders may turn files, cameras, databases, or notebook arrays into the
same request. Loading remains outside the service.

## ROI result lifecycle

`run_roi_accumulation` validates the request, emits synchronous progress,
checks cancellation, calls `iclotspython.core.accumulation`, and returns an
immutable `ROIAccumulationResult`.

The result contains compatibility-baseline labels/time, channel area and
signed-accumulation series, units, parameters, structured warnings,
provenance, and a sanitized suggested export stem. It contains no Tkinter
objects, pandas DataFrames, figures, open handles, callbacks, or mutable GUI
state.

Presentation helpers provide:

- `roi_table`: immutable columns and row tuples;
- `roi_export_records`: format-neutral row mappings;
- `roi_plot_series`: named x/y series without creating figures.

Phase 3B exporters and plotters may consume these structures. They must not
move scientific calculations out of the core.

## Validation, warnings, and errors

Invalid inputs use stable error codes and fields. Unsupported array shape,
scientific-core failure, internal orchestration failure, and cancellation have
distinct categories. Every `ApplicationError` carries a user message,
developer detail, and optional field/remediation.

Normal zero signal is a successful result. Warnings are structured records
rather than display strings. Current warning codes cover:

- artificial compatibility baseline;
- autogenerated labels or time values;
- empty signal by channel or across the request;
- non-monotonic caller times;
- thresholds outside the observed value range.

Interfaces decide how to display warnings. The service never opens a dialog.

## Progress and cancellation

`ProgressEvent` records workflow ID, stable stage, current and total steps,
monotonic completion fraction, message, and optional frame label.
`ProgressCallback` and `CancellationCheck` are lightweight protocols.

The ROI service is synchronous. It emits validation start/completion, one
processing event per frame and selected channel, assembly, and completion.
Cancellation raises `WorkflowCancelled` with code `workflow_cancelled`.
Qt threads, workers, and event loops are intentionally not part of Phase 3A.

## Provenance and reproducibility

ROI provenance records the workflow and contract versions, an explicit
unversioned application placeholder, scientific-core module, parameters,
channel convention, selected channels, thresholds, calibration, frame count,
time/baseline conventions, caller source identifiers, caller metadata, and
warning codes.

Provenance collection does not invoke Git, inspect the environment, or collect
personal information. Version discovery may be added later through an
explicit, testable release mechanism.

## Transitional legacy integration

The supported transitional architecture is:

```text
Tkinter adapter  ---> scientific core
future UI        ---> application service ---> scientific core
```

`analysis/occroi.py` remains on its proven Phase 2 direct-core path. It was not
rewired for Phase 3A. A later migration may route it through the application
service only after DataFrame, plotting, export, dialog, and presentation
parity is independently established.

## Intended consumers and separation

PySide6 is the leading future desktop choice. A PySide6 screen can construct
requests, translate structured errors/warnings, subscribe to progress, and
bind table/plot series without knowing legacy internals. No PySide6 dependency
is introduced here.

The same contracts may later support notebooks or a CLI, but neither is
implemented or designed as a separate runtime in Phase 3A.

Plotting and exporting remain separate because they involve presentation
policy, file formats, user destinations, and optional dependencies. The
application service remains deterministic, headless, and free of filesystem
side effects.

## Phase 3B output-service boundary

Phase 3B adds `iclotspython.output` as the only modern layer in this
architecture that writes presentation artifacts:

```text
ROIAccumulationResult
        |
        +--> PlotRequest   --> render_roi_plot --> PNG
        |
        +--> ExportRequest --> export_roi_data  --> CSV / XLSX
                                      |
                                      v
                               OutputManifest
```

`ROIAccumulationResult` is the only numerical input. Output services consume
`roi_table` and `roi_plot_series`; they do not import the scientific core,
threshold arrays, calibrate values, or calculate accumulation.

Requests explicitly define destination directory, optional directory
creation, deterministic filename/stem, requested formats, overwrite policy,
and multi-file failure policy. The default overwrite policy is
`FAIL_IF_EXISTS`. Existing files are never changed unless the caller selects
`REPLACE`. Explicit replacement is recorded in the manifest.

Destination directories must already exist unless `create_destination=True`.
Invalid or unwritable destinations fail before any output target is
attempted. Filenames are leaf names with format-appropriate suffixes; callers
cannot smuggle a second destination through a filename.

Every requested target receives an `OutputFileRecord` with created, replaced,
failed, or not-attempted status. Successful records include size and SHA-256.
Warnings and failures use structured `OutputIssue` values. Overall manifests
report success, partial success, or failure. Manifests stay in memory and do
not create an unrequested manifest file.

Multi-format export supports `STOP` and `CONTINUE`. `STOP` marks later formats
not attempted after a failure. `CONTINUE` permits successful siblings and
returns a partial manifest. Successfully completed files are not silently
rolled back because the manifest is the explicit record of partial output.
Temporary files used for explicit replacement are removed on failure.

PNG rendering uses matplotlib's object-oriented `Figure` with the
non-interactive Agg canvas; it does not import `pyplot`. CSV uses the standard
library. XLSX uses openpyxl directly. Matplotlib and openpyxl imports are lazy,
so importing `iclotspython.output` loads no GUI toolkit, plotting backend,
pandas, or spreadsheet engine.

The supported modern boundary is now:

```text
future UI / CLI / notebook
        |
        v
application service --> ROIAccumulationResult
                              |
                              v
                      output services --> files + manifest
```

File selection, user confirmation, progress UI, and display of manifest
issues remain responsibilities of future consumers. Phase 3B does not add a
GUI, CLI, notebook wrapper, background worker, or project/session model.

## Future scope boundaries

Phase 3A does not include PySide6 windows, Qt Designer files, file pickers,
project/session persistence, batch processing, CLI or web APIs, packaging,
installers, scientific corrections, all-workflow migration, or Tkinter
removal. Phase 3B adds only headless ROI plot/CSV/XLSX output services; all
other items remain out of scope.
